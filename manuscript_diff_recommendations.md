## 3. 生成器関連

### 3.1 Location: §5.3 環境の生成、第一段落(生成手順の第二ステップ)

```diff
- 第二に、設計図を r 件の中核文書(議事録・答申・内部メモ等の類型)に符号化し、結合十分性と最小性を機械的に検査する。
+ 第二に、設計図を r 件の中核文書(議事録・答申・内部メモ等の類型)に符号化する。基準実装では Y₀ の成分数を K = r とし、各中核文書が Y₀ のちょうど一成分を排他的に担う一対一対応を採る——この規約が結合十分性・最小性と、仮定4の線形な回収—歪み写像を同時に、構成的に強制する。対応の成立は生成のたびに機械的に検査する。
```

Rationale: 実装は K = r の一対一対応で仮定4を設計時に強制しており(各中核文書は自成分の値を含み、他成分の値を含まないことを検査)、本文の「機械的に検査する」より強い保証を与えている。仮定4の明文化(v0.4 改訂 A3)と §5.3 を明示的に接続する。

### 3.2 Location: §5.3 環境の生成、第二段落(生成器の決定性)

```diff
- 生成器は決定的な骨格(設計図 → 文書計画 → テンプレート充填)を基本とし、LLM は表層の文体化にのみ用いる。
+ 生成器は決定的な骨格(設計図 → 文書計画 → テンプレート充填)を基本とし、基準実装は (q, r, c, seed) を与えれば完全に決定的で、LLM を一切用いない。LLM による表層の文体化は、生態学的妥当性を高めるための任意の追加層として主実験前に導入し、その場合のみ C.7 の開示を適用する。理論命題の検定(表1)は決定的生成のみで完結する。
```

Rationale: 現時点の実装はテンプレートのみで LLM 未使用。本文が「LLM を用いた」と読める余地を消し、AI 利用開示の適用条件を正確にする。理論検定が LLM 生成に依存しないことは、検証の頑健性としてむしろ利点である。
